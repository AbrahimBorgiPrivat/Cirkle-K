Repository Structure

The repository is split so that documentation, runtime assets, code, PBIP workspaces, and the static site each have a clear role.

Why the layout helps

  • Docs, data assets, code, workspaces, and site files each have a clear home.
  • The structure makes the solution easier to rerun, review, and explain.
  • Technical assets stay close to the code and models that depend on them.

Main folders

  • `docs` holds the original interview and simulation documents used for project framing and conclusions.
  • `resource` contains CSV input, JSON input, and Power BI themes or images.
  • `source/code` contains the Python package, runtime definitions, and Docker services.
  • `source/workspaces` contains the PBIP models plus Tabular Editor automation assets.
  • `site` contains the GitHub Pages-ready static documentation site.

Why this split works

  • Input data is separated from execution code.
  • PBIP workspaces stay versioned independently from Docker services.
  • The static site can be published without exposing the full working environment.
  • Tabular assets live next to the workspaces they update.

Operational view

  • Docker services build and load the reporting layer.
  • PBIP and Tabular assets define the analytical layer.
  • The site and README define the presentation layer.
Good fit for GitHub

The layout keeps large static visuals in `site/assets`, technical scripts in `source`, and project narrative in `docs` and the static site.
