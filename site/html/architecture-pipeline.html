Article 02 / Pipeline

Pipeline

Both cases are built as reproducible flows, but at different scales. The interview case is compact and SQL-first. The simulation case is broader and service-based.

What this page shows

  • The interview flow is a short path from CSV inputs to PostgreSQL and a PBIP model.
  • The simulation flow adds Docker services, synthetic generation, and reporting-table orchestration.
  • Both setups aim for a repeatable route from source data to semantic reporting.
Interview flow

CSV to PostgreSQL to PBIP

The interview case loads CSV files into PostgreSQL, applies SQL structure, and keeps Power BI focused on semantic logic and visuals.

Simulation flow

Services and orchestration

The simulation case combines external Danish data, seed files, synthetic generation, upserts, and Dockerized services before the report layer.

Interview pipeline

  • CSV facts and dimensions are loaded into PostgreSQL.
  • SQL views shape the reporting layer.
  • The PBIP model uses a clean star schema and grouped measures.

Simulation pipeline

  • Python generates customers, cards, campaigns, transactions, and cashiers.
  • DAR and DAGI data enrich the geographic side of the model.
  • Docker services handle ingestion, load, simulation, and reporting tables.
Implementation style

Runtime-driven services and PBIP workspaces are kept in source control, while static input data is mounted into containers when relevant.
